# 整合性チェック → コミット（オプションで push / PR）

作業ツリーの変更を、**変更全体の整合性を検証してから**きれいにコミットするためのスキル。
コード・設計ドキュメント・設定など対象は問わない。「複数ファイルにまたがる変更の伝播漏れを潰してからコミットする」という頻出パターンを固定化したもの。

**既定はローカルコミットまで。** push / PR 作成は $ARGUMENTS でオプトインしたときだけ実行する。

## オプション（$ARGUMENTS）

フラグは省略可。複数指定可。フラグ以外のテキストはブランチ名やコミット分割方針などの補足として扱う。

| フラグ | 挙動 |
|--------|------|
| （なし） | コミットのみ。push も PR もしない（**既定**）。 |
| `--push` | コミット後、現在のブランチを origin に push する。 |
| `--pr` | コミット後、push して PR を作成する（`--push` を含意）。PR は既定で **Draft（下書き）**。ただし対象ブランチに既に open な PR があれば新規作成せず、**本文を最新のコミット内容で上書き更新する**。 |
| `--ready` | `--pr` と併用したとき、PR を **通常PR（レビュー可 / Ready for review）** で作成する。 |

> **Draft と 通常PR の違い**: どちらも同じ PR の状態。Draft は「作業中・レビュー依頼前」でマージ不可・レビュー自動リクエストなし。通常PR（レビュー可）は最初からレビュー・マージ可能。あとから GitHub 上や `gh pr ready` で切り替えられる。

## スコープ

- 対象は作業ツリーの未コミット変更（staged / unstaged 両方）。
- コミットメッセージ・PR は日本語（英語指定がある場合を除く）。

## 手順

### Step 1: 変更内容の把握とオプション解釈

```bash
git status
git diff HEAD
git branch --show-current
git log --oneline -5
```

- $ARGUMENTS を解釈し、push / PR を行うか（`--push` / `--pr` / `--draft`）を確定する。
- 変更ファイル一覧と差分を確認し、**何を・なぜ変えたか**を要約する。
- 変更が「単一の論点」か「複数の無関係な論点の混在」かを見極める（Step 4 の分割判断に使う）。

### Step 2: 整合性チェック

変更が**複数ファイルにまたがる、またはリネーム・移動・シグネチャ変更・用語変更を含む**場合、変更ファイルだけでなく**関連する範囲全体**に対して次を検証する（該当しなければスキップ）。

- **伝播漏れ**: 変更した名前（関数・変数・型・エクスポート、テーブル名・カラム名、識別子・用語、見出しなど）の**旧名が他の箇所に残っていないか** `git grep` で確認する。
- **参照の破損**: 変更に伴い壊れた参照がないか確認する。
  - コード: import / export、呼び出し箇所、型参照。
  - ドキュメント: 内部リンク（`[...](...)` や `#見出し`）、相互参照。
- **表記ドリフト**: 同じ概念に新旧の表記が混在していないか確認する。

```bash
# 旧名・旧用語が残っていないかリポジトリ全体を検索する例
git grep -n "<旧名や旧用語>"
```

- 漏れ・破損が見つかったら**コミット前に修正する**。勝手に新しい名前・用語を創作せず、合意済み／変更後の表記のみを伝播させる。
- 検証結果（チェックした名前と、修正した箇所）を短くメモし、Step 6 のレポートに載せる。

### Step 3: main 上なら作業ブランチを切る

現在のブランチが `main`（デフォルトブランチ）の場合、コミット前に作業ブランチを作成する。**main に直接コミット・push しない。**

```bash
git switch -c <ブランチ名>
```

- ブランチ名は既存慣習に合わせる（例: `docs/<スラッグ>`, `feat/<スラッグ>`, `fix/<スラッグ>`）。$ARGUMENTS に指定があればそれを優先する。
- すでに feature ブランチ上ならそのまま使う。

### Step 4: コミット作成

- **原則、論理単位でまとめる。** 単一の論点なら1コミット。無関係な変更が混在している場合は論理単位で分割する（それぞれ関連ファイルをまとめて add）。
- **subject**: Conventional Commits 形式の日本語1行（`feat:` / `fix:` / `docs:` / `refactor:` / `chore:` など）。
- **body**: subject だけで意図が伝わらない場合に、何をなぜ変えたかを追記。
- **footer**: コミットメッセージの末尾に必ず次を付ける。

  ```
  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  ```

```bash
git add <変更ファイル>
git commit -m @'
<type>: <日本語の要約>

（必要なら本文）

Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
'@
```

### Step 5: push / PR（オプション指定時のみ）

`--push` も `--pr` も指定されていなければ、この Step はスキップして Step 6 へ。

**push（`--push` または `--pr`）**

```bash
git push -u origin <現在のブランチ>
```

- 履歴が乖離していて通常 push が失敗する場合は**止めてユーザーに確認する**。`--force` は使わない。

**PR 作成（`--pr` のときのみ、push 成功後）**

まず対象ブランチに **既に open な PR がないか確認する**。

```bash
gh pr list --head <現在のブランチ> --state open --json number,url,isDraft
```

- **既に open な PR があれば新規作成しない。** push 済みのコミットは自動でその PR に反映される。加えて、**本文を最新のコミット内容で上書き更新する**（下記の本文構成に従い `gh pr edit <番号> --body` で置き換える）。
  - **上書きするのは本文のみ。** タイトル・Draft/通常の状態・assignee は変更しない。
  - 手で書き加えた記述も本文ごと置き換わる点に注意（それが今回の指定挙動）。

  ```bash
  gh pr edit <番号> --body @'
  ## 背景・目的
  ...

  ## 変更内容
  - ...

  ## 補足
  ...

  🤖 Generated with [Claude Code](https://claude.com/claude-code)
  '@
  ```
- open な PR がなければ、以下で新規作成する。
  - PR タイトル: コミット subject（Conventional Commits 形式の日本語）。
  - ベースブランチ: `main`（デフォルトブランチ）。
  - **assignee に自分を追加する**（`--assignee @me`）。
  - **既定は Draft（下書き）**（`gh pr create --draft`）。`--ready` 指定時のみ通常PR（レビュー可）で作成する。
  - PR 本文は次の構成に従う。

  ```markdown
  ## 背景・目的

  （なぜこの変更が必要かを簡潔に）

  ## 変更内容

  - （何を変えたかを箇条書きで）

  ## 補足

  （任意。仮の命名・オープン論点・レビュー時の注意など）

  🤖 Generated with [Claude Code](https://claude.com/claude-code)
  ```

```bash
# 既定は --draft を付ける。--ready 指定時のみ外す。assignee に自分を追加する。
gh pr create --base main --draft --assignee @me --title "<subject>" --body @'
## 背景・目的
...

## 変更内容
- ...

## 補足
...

🤖 Generated with [Claude Code](https://claude.com/claude-code)
'@
```

### Step 6: レポートと次アクション

以下を報告する。

- 作成したコミット（ハッシュ・subject）と、コミットした／していないファイル。
- Step 2 の整合性チェック結果（検証した名前・用語と、修正した箇所。問題なしなら「参照漏れなし」。実施していなければ「対象外」）。
- 現在のブランチ名。push した場合はその旨。
- PR を作成した場合は URL・Draft/通常(レビュー可) の別・assignee。既存の open PR があった場合は、本文を上書き更新した旨と URL。
- push / PR を行っていない場合は、次アクション（手動 push / PR、または `/github:pull-request-review` 等）を案内する。

## 注意事項

- **既定は push も PR もしない。** `--push` / `--pr` が指定されたときだけ行う。
- **main に直接コミット・push しない**（Step 3）。PR のベースは常に `main`。
- `git commit --amend` や `git push --force` などの履歴改変は行わない。履歴が乖離していたら止めてユーザーに確認する。
- `--no-verify` でフックをスキップしない。フックが失敗したら原因を調べて直す。
- コミットメッセージ・PR は日本語で書く（英語指定がある場合を除く）。
- 整合性チェックで**新しい名前・用語を勝手に作らない**。合意済み／変更後の表記のみを伝播させる。
