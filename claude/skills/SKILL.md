# ドキュメント整形スキル（サマリ先頭・図解多用・比喩つき）

ドキュメントを **決まったフォーマット** で生成するためのスキル。読み手が「冒頭のサマリだけで全体像と目的をつかめ、本文で図と箇条書きを追いながら細部を理解できる」状態を必ず作る。技術資料・設計書・解説記事・README など、対象は問わない。

書き始める直前に、必ず次の2ファイルを読むこと。

- 文書の雛形: [reference/template.md](reference/template.md)
- mermaid 図の選び方・記法: [reference/mermaid-guide.md](reference/mermaid-guide.md)

## 使うとき / 使わないとき

- **使う**: まとまった文書を新規に書き起こすとき（資料・ドキュメント・解説・まとめ・設計書・README 等）。
- **使わない**: 既存文書への一行追記や誤字修正、コードの実装・デバッグ、コミット/PR 本文の作成。これらは通常どおり対応する。
- 迷ったら「読み手向けにまとまった説明を新規に書くか？」で判断する。Yes ならこのフォーマットに従う。

## 出力フォーマット（必ずこの構成にする）

文書は常に **「1. 冒頭サマリ」→「2. 本文セクション群」** の順で構成する。

### 1. 冒頭サマリ（最重要・省略不可）

文書の一番上に置く。以下の3要素を **この順で** 必ず含める。

- **1-1. この資料の目的** — 「この資料を読むと何が理解できるか」を明確に書く。箇条書きで「読者」「読み終えたときに分かること（3点前後）」を示す。網羅的な説明はここではしない。
- **1-2. 全体像の図（mermaid）** — 資料全体の構造・登場人物・流れを1枚の mermaid 図で俯瞰できるようにする。この図は「本文の地図」であり、以降のセクションがこの図のどこを説明しているか対応づく状態にする。図の種類の選び方は reference/mermaid-guide.md を参照。
- **1-3. ひとことで言うと（比喩）** — 中学生でも分かるような **身近なものへの比喩** で全体像を1〜3文で説明する。専門用語を使わず「要するに〇〇のようなもの」と言い切る。比喩の作り方は後述。

### 2. 本文セクション群

- **セクションはできるだけ細かく分割する。** 1セクション＝1トピックを目安にし、長い説明は複数セクションに割る。
- **各セクションに図か箇条書きを最低1つ入れる。** 文章の塊で説明せず、mermaid 図・表・箇条書きへ積極的に分解する。同じ内容なら図＞箇条書き＞地の文の優先度で表現する。
- **2-1. サブセクションを活用する。** 1つのセクションが複数の論点を含むなら `###` / `####` でサブセクションに割ってよい。階層で構造を見せる。
- mermaid は1種類に固執せず、内容に合うもの（フロー/シーケンス/状態/クラス/ER/ガント等）を使い分ける。

## 作業フロー

### Step 1: 骨子設計

1. 引数・文脈から **テーマ・読者層・出力先** を確定する（曖昧ならユーザーに確認）。
2. 伝えるべき論点を洗い出し、**セクション構成（見出しの一覧）** を先に決める。細かく割る方針で。
3. 冒頭サマリの 1-2 に置く「全体像の図」で何を俯瞰させるかを決める。

### Step 2: 冒頭サマリの執筆

reference/template.md の雛形に沿って、1-1（目的）→ 1-2（全体像 mermaid）→ 1-3（比喩）の順に書く。ここだけ読めば全体像が分かる状態にする。

### Step 3: 本文の執筆

Step 1 で決めた見出しに沿って各セクションを書く。各セクションで「この内容は図・表・箇条書きにできないか？」を必ず自問し、できる限り図解・リスト化する。必要ならサブセクションへ分割する。

### Step 4: セルフレビューと出力

末尾のチェックリストで自己点検し、不足があれば Step 2/3 に戻る。mermaid はレンダリング可能な構文か確認する。指定の出力先（なければユーザーに確認、もしくは会話に出力）へ書き出す。

**出力ファイル名は英語（半角）にする。** 内容が日本語でもファイル名は英語の kebab-case（例: `aws-architecture-overview.md`, `agentcore-migration-guide.md`）にし、日本語・全角文字・空白を使わない。拡張子は原則 `.md`。ユーザーがファイル名を明示した場合はそれに従う。

## 比喩の書き方（1-3 のため）

- **身近な題材に置き換える**：宅配便、料理、学校、電車、図書館、受付窓口、交通整理など、中学生の生活圏にあるもの。
- **専門用語を1つも使わない。** 使いたくなったら、それを比喩の中の役割に翻訳する（例: キュー→「レジの行列」、ロードバランサ→「客を空いている窓口へ案内する係」、キャッシュ→「よく使う道具を手元に置いておく」）。
- **1〜3文で言い切る。** 長い比喩は逆に分かりにくい。「要するに〇〇は、△△における□□のようなもの」の型が使いやすい。
- 正確さより **直感的な理解** を優先する。細かい例外は本文で補う。

具体例は reference/template.md の「比喩の例」を参照。

## mermaid の使い分け（要点）

- 全体像・処理の流れ → **flowchart**
- 時系列のやり取り（API・コンポーネント間） → **sequenceDiagram**
- 状態の遷移 → **stateDiagram-v2**
- データ構造・テーブル関係 → **erDiagram** / **classDiagram**
- スケジュール・工程 → **gantt**

詳細な記法と選択基準は reference/mermaid-guide.md を参照。

## セルフレビューチェックリスト

- [ ] 冒頭にサマリがあり、**目的 → 全体像図 → 比喩** の3点が揃っているか
- [ ] 目的は「読むと何が分かるか」を具体的に示しているか
- [ ] 全体像の mermaid 図があり、本文の構成と対応づくか
- [ ] 比喩は中学生に伝わる身近さで、専門用語を使っていないか
- [ ] セクションは十分に細かく割れているか（1セクション＝1トピック）
- [ ] 各セクションに図・表・箇条書きのいずれかが入っているか
- [ ] 複数論点のセクションはサブセクションに分割したか
- [ ] すべての mermaid がレンダリング可能な構文か
- [ ] 出力ファイル名は英語の kebab-case か（日本語・全角・空白を含まない）

## 行動指針

- すべて日本語で書く（英語指定がある場合を除く）。
- **図・箇条書きファースト**。地の文の長文は避け、構造で見せる。
- 内容の正確さを損なわない範囲で、比喩とサマリで「まず全体像」を掴ませることを最優先する。
- フォーマットは固定だが、テーマに応じて図の種類・セクション粒度は柔軟に選ぶ。
